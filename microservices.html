0.3 Microservices

Microservices use web services (REST/SOAP) for communication. A web service is simply a medium for communication.

REST Web Services

Representational State Transfer

  • Architectural style (not a protocol)
  • Uses HTTP (GET, POST, PUT, DELETE)
  • Usually returns JSON
  • Lightweight & fast
  • Stateless

SOAP Web Services

Simple Object Access Protocol

  • Protocol-based: SOAP is a protocol, not just a style.
  • Transport-independent: Can use HTTP, SMTP, TCP, etc.
  • Uses XML: SOAP messages are strictly XML formatted.
  • Operations via WSDL (Web Services Description Language).
  • Heavyweight: Formal, strict standards, slower than REST.
  • Can be Stateful or Stateless.
  • Built-in error handling & security (WS-Security).

1 What is Microservice?

"Microservices is an architectural style to design our application uniquely with several small modules developed individually, packaged individually, and deployed individually in individual processes."

2 Why Microservice?

Why did microservices come into the picture when we were previously developing and deploying our application in an efficient way using the monolithic approach? Let's explore the previous architecture and its disadvantages.

3 What is monolithic architecture?

We develop all services together and then package all services in a single WAR file and then deploy it to the server.

4 Drawbacks of monolithic architecture?

SHORTHAND: PCAEC

Performance Issues

Due to huge data being packaged in a single WAR file, performance is impacted.

Continuous Deployment

Not possible; if one module is not working, the whole project is disturbed, resulting in long wait times.

Technological Rigidity

Adapting new technologies is difficult when modules must match (e.g., all Java or all Python).

Overloaded Resources

Overloaded IDE and web container due to the massive codebase size.

Exception Propagation

Propagation is not proper; one module failing often impacts all others.

Code Readability

Working with one module requires an understanding of all others.

5 Advantages to use micro service?

Performance & Registry

As each service is deployed individually in different containers (using registries like Eureka Service Registry), performance won’t be slow. Quick development and delivery make it easy to track and fix bugs.

Continuous Deployment

Continuous deployment will not suffer. If you want to deploy a bug fix in one component, you just change and deploy that specific module without bothering other services already registered in the Service Registry.

Technology Diversity

Adopting new technologies or languages is very easy, as you can implement a different language for each module of the application.

Fast Startup & Auto Restart

As each service uses separate WAR/JAR files with less data, server start-up is fast. We can run multiple ports and use DevTools or Spring Loader for automatic restarts on logic changes.

Proper Exception Propagation

Exception propagation is proper; if any service fails, it won’t impact the entire application. We can propagate it using Hystrix.

Clear Code Readability

Code readability is very clear. Suppose a new resource joins and needs to work on the Shipping microservice; they can easily check the flow of that specific individual service.
