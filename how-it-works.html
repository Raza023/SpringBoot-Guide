How Spring Boot Internally Works

Unveiling the magic behind the "Convention over Configuration" philosophy.

Auto-configuration Magic

Pre-configured Jars

We use pre-configured jars provided by Spring Boot developers. Instead of manual setup, these jars come with sensible defaults.

Starter POMs

  • spring-boot-starter-web
  • spring-boot-starter-data-jpa
  • spring-boot-starter-test

Adding these "starters" transitively brings in all required dependencies (jars).

spring.factories

After adding the starter pom, the core registry is META-INF/spring.factories. It contains a list of Auto-Configuration classes.

"Spring Boot reads this file at startup to find @Configuration classes that provide the 'magic' beans."

The Decision Logic: @Conditional

Before loading a configuration, Spring Boot performs a "Conditional Check":

  • @ConditionalOnClass
  • @ConditionalOnProperty
  • @ConditionalOnMissingBean
  • @ConditionalOnResource
IF CONDITIONS ARE MET Enabled

Logic Example: DispatcherServlet

Scenario A

If we have spring-boot-starter-web jar → found @ConditionalOnClass(Servlet.class) → it enables DispatcherServlet automatically.

Scenario B

If we don't have the web starter → condition fails → it skips DispatcherServlet, keeping the app lightweight.

The Startup Lifecycle

@SpringBootApplication

This single annotation represents a combination of three powerful traditional Spring annotations:

@Configuration
@EnableAutoConfiguration
@ComponentScan

Note: The main method is primarily for IDE launching; in production, deployment scenarios vary.

SpringApplication.run() Pipeline

SpringApplication.run() helps us to start the ApplicationContext that searches the classes with @Configuration, @Component, @Service and it will initialize all the declared beans in those Configuration classes and initiate the required beans. And stores in IOC (Inversion of Control) (IoC container is a framework for implementing automated dependency injection.) of JVM
